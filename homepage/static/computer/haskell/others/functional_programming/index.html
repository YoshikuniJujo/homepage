「関数型プログラミング」について

はじめに

「関数型プログラミング」という言葉が注目されているようだ。 Haskellerとしてはこの言葉を聞いて無関心ではいられない。 どうにもこの言葉は定義のはっきりしない言葉であり いろいろと物議をかもしているようだ。 いろいろなサイトを参考にしつつ、自分なりの意見をまとめてみようと思う。

9月18日のメール: 「すし」のたとえ

追記です。

あるブログで「...を理解しただけでは関数型言語を理解したことにはならない」ということを「カリフォルニアロールを食べても寿司を理解したことにはならない」というたとえで説明していました。

関数型言語の定義をお寿司の定義にたとえるのはなかなかいいように思います。お寿司の定義はなかなか難しいです。まず「握り寿司」はまさに「寿司のなかの寿司」です。「ザ・寿司」です。これがHaskellにあたります。しかし歴史をたどればもともとは「魚を塩と飯で漬け込み熟成させたもの」でした。これが、まあLispでしょう。

「稲荷寿司」は寿司でしょうか、「ちらし寿司」は寿司でしょうか、「手巻寿司」はどうでしょうか。これらの「寿司っぽさ」は「握り寿司」よりも低いです。「お寿司食べたい」と言われてこれらを出したとします。このときの反応は人によってさまざまでしょう。「押し寿司」というものもありますね。

これらの多くに共通する特徴として「酢飯」があります。そこである人がこう定義します。「お寿司とは酢飯を使った食べ物である」と。しかしコンビニおにぎりは酢飯を使っているようですが、これはお寿司ではありません。また家庭の「手巻き寿司」は手抜きして普通のご飯を使うかもしれない。それでも「寿司」と呼ばれるかもしれない。

「寿司」とは何か?が定義できないように「関数型」とは何か?も定義できません。また、定義は時代とともに変化します。ただ、最も寿司っぽい「握り寿司」から、あまり寿司っぽくない「カリフォルニアロール」まで寿司っぽさ」のグラデーションがあり、そのグラデーション上の位置づけについては、ある程度のコンセンサスがあるかとは思います。

「この言語は関数型言語か」「この手法は関数型と呼べるか」という議論は「稲荷寿司は寿司か」といった議論と同じことかと思います。ある人は「寿司ってついてるんだから寿司だ」と言いますが、ある人は「じゃあお寿司食べに行こうって言われたときに稲荷寿司で納得するか」と言い返します。

ただ「握り寿司」が文句なしに「寿司」であるようにHaskellは文句なしに「関数型言語」です。そこらへんのコンセンサスはあるかと思います。

また、より厳密な言葉を使おうとしたら個々の特徴ごとに

  • 参照透過性
  • 第一級関数
  • 遅延評価
  • (弱)頭部正規形までの評価
  • 静的型付け

のように個々の特徴に分ければ、これはもう厳密に定義されています。
